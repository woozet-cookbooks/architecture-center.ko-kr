## <a name="scalability"></a><span data-ttu-id="14345-117">확장성</span><span class="sxs-lookup"><span data-stu-id="14345-117">Scalability</span></span>

<span data-ttu-id="14345-118">확장성은 증가된 부하를 처리하는 시스템의 기능입니다.</span><span class="sxs-lookup"><span data-stu-id="14345-118">Scalability is the ability of a system to handle increased load.</span></span> <span data-ttu-id="14345-119">응용 프로그램은 다음 두 가지 주요 방법으로 확장될 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-119">There are two main ways that an application can scale.</span></span> <span data-ttu-id="14345-120">예를 들어, 수직적 크기 조정(*강화*)이란 더 큰 VM 크기를 사용하여 리소스의 용량을 늘리는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="14345-120">Vertical scaling (scaling *up*) means increasing the capacity of a resource, for example by using a larger VM size.</span></span> <span data-ttu-id="14345-121">수평적 크기 조정(*확장*)이란 VM 또는 데이터베이스 복제본과 같은 리소스의 새 인스턴스를 추가하는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="14345-121">Horizontal scaling (scaling *out*) is adding new instances of a resource, such as VMs or database replicas.</span></span> 

<span data-ttu-id="14345-122">수평적 크기 조정에는 수직적 크기 조정에 비해 많은 장점이 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-122">Horizontal scaling has significant advantages over vertical scaling:</span></span>

- <span data-ttu-id="14345-125">수평적 크기 조정은 탄력적입니다.</span><span class="sxs-lookup"><span data-stu-id="14345-125">Horizontal scale is elastic.</span></span> <span data-ttu-id="14345-126">부하가 증가하면 더 많은 인스턴스를 추가하거나 더 조용한 기간 동안 제거할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-126">You can add more instances if load increases, or remove them during quieter periods.</span></span>
- <span data-ttu-id="14345-127">일정에 따라 또는 부하의 변화에 대한 응답으로 확장이 자동으로 트리거될 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-127">Scaling out can be triggered automatically, either on a schedule or in response to changes in load.</span></span> 
- <span data-ttu-id="14345-128">확장은 강화보다 저렴할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-128">Scaling out may be cheaper than scaling up.</span></span> <span data-ttu-id="14345-129">여러 작은 VM을 실행하면 큰 단일 VM보다 비용이 저렴할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-129">Running several small VMs can cost less than a single large VM.</span></span> 
- <span data-ttu-id="14345-130">수평적 크기 조정은 중복성을 추가하여 복원력을 향상할 수도 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-130">Horizontal scaling can also improve resiliency, by adding redundancy.</span></span> <span data-ttu-id="14345-131">인스턴스가 중지되면 응용 프로그램이 계속 실행됩니다.</span><span class="sxs-lookup"><span data-stu-id="14345-131">If an instance goes down, the application keeps running.</span></span>

<span data-ttu-id="14345-132">수직적 크기 조정의 장점은 응용 프로그램을 변경하지 않고도 수행할 수 있다는 점입니다.</span><span class="sxs-lookup"><span data-stu-id="14345-132">An advantage of vertical scaling is that you can do it without making any changes to the application.</span></span> <span data-ttu-id="14345-133">하지만 특정 시점에 더 이상을 강화할 수 없는 제한에 도달할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-133">But at some point you'll hit a limit, where you can't scale any up any more.</span></span> <span data-ttu-id="14345-134">이 시점에서 추가 크기 조정은 수평적이어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="14345-134">At that point, any further scaling must be horizontal.</span></span> 

<span data-ttu-id="14345-135">시스템에 수평적 크기 조정이 포함되어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="14345-135">Horizontal scale must be designed into the system.</span></span> <span data-ttu-id="14345-136">예를 들어 VM을 부하 분산 장치 뒤에 배치하여 확장할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-136">For example, you can scale out VMs by placing them behind a load balancer.</span></span> <span data-ttu-id="14345-137">그러나 풀의 각 VM은 모든 클라이언트 요청을 처리할 수 있어야 합니다. 따라서 응용 프로그램이 상태 비저장이거나 상태를 외부적으로(즉, 분산된 캐시에서) 저장해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="14345-137">But each VM in the pool must be able to handle any client request, so the application must be stateless or store state externally (say, in a distributed cache).</span></span> <span data-ttu-id="14345-138">PaaS 관리 서비스는 수평적 크기 조정 및 자동 크기 조정이 기본 제공됩니다.</span><span class="sxs-lookup"><span data-stu-id="14345-138">Managed PaaS services often have horizontal scaling and auto-scaling built in.</span></span> <span data-ttu-id="14345-139">이러한 서비스를 크기 조정하는 용이성이 PaaS 서비스를 사용하는 주요 이점입니다.</span><span class="sxs-lookup"><span data-stu-id="14345-139">The ease of scaling these services is a major advantage of using PaaS services.</span></span>

<span data-ttu-id="14345-140">하지만 더 많은 인스턴스를 추가하는 것은 응용 프로그램의 크기 조정이 아닙니다.</span><span class="sxs-lookup"><span data-stu-id="14345-140">Just adding more instances doesn't mean an application will scale, however.</span></span> <span data-ttu-id="14345-141">다른 곳에 병목 상태를 푸시할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-141">It might simply push the bottleneck somewhere else.</span></span> <span data-ttu-id="14345-142">예를 들어 웹 프런트 엔드의 크기를 조정하여 추가 클라이언트 요청을 처리하는 경우 데이터베이스에서 잠금 경합을 트리거할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-142">For example, if you scale a web front-end to handle more client requests, that might trigger lock contentions in the database.</span></span> <span data-ttu-id="14345-143">그러면 데이터베이스에 대한 처리량을 높일 수 있도록 낙관적 동시성 또는 분할 데이터와 같은 추가 조치를 사용해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="14345-143">You would then need to consider additional measures, such as optimistic concurrency or data partitioning, to enable more throughput to the database.</span></span>

<span data-ttu-id="14345-144">항상 성능 및 부하 테스트를 구축하여 이러한 잠재적인 병목 상태를 찾습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-144">Always conduct performance and load testing to find these potential bottlenecks.</span></span> <span data-ttu-id="14345-145">데이터베이스와 같은 시스템의 상태 저장 파트는 가장 일반적인 병목 현상의 원인이며 수평적으로 규모를 조정하도록 신중하게 디자인해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="14345-145">The stateful parts of a system, such as databases, are the most common cause of bottlenecks, and require careful design to scale horizontally.</span></span> <span data-ttu-id="14345-146">하나의 병목 상태를 확인하면 다른 곳에서도 병목 상태를 발견할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="14345-146">Resolving one bottleneck may reveal other bottlenecks elsewhere.</span></span>
